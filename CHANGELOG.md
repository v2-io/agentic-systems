# CHANGELOG

The forward-going record of substantive changes to the framework. Entries are ordered most-recent-first, dated, and grouped by theme within a date.

## How this file relates to TODO.md, LOG.md, audits/, and git

- **[TODO.md](TODO.md)** — current tactical work items (active findings, queued spike follow-ups, deferred items, open MEDIUM items, missing segments).
- **CHANGELOG.md** *(this file)* — consolidated narrative of substantive changes from 2026-04-24 onward. Cycle-shape narratives ("what changed about how the theory thinks", structural moves, conventions adopted) go here. Entry granularity is the day; commit hashes appear inline for traceability.
- **[LOG.md](LOG.md)** — frozen archaeology of cycles through 2026-04-24. Read it when the *origin* of a current commitment matters: whether a "settled" item rests on derivation or on a cycle's working consensus, or how an audit finding's prior status should shape current routing.
- **`audits/pending-findings-YYYY-MM-DD.md`** — original audit-finding records (per-cycle burden-of-proof writeups and resolution trails).
- **Git** — primary source for "what changed in code/segments." Use `git log --follow` for slug-renamed segments.

CHANGELOG carries the *narrative shape* of each cycle (what conventions changed, what disciplines emerged, what the cycle was about); the per-cycle `audits/pending-findings-*` files carry the *checklist trail* (which findings were verified, what fixes landed). They overlap in scope but serve different reading needs.

**New project-level naming principle landed: Criterion 9 — standalone citability** (`doc/naming-principles.md`). The principle articulates a test the file didn't previously formalize: *"Could a researcher reading or referencing this name in a context outside the project — in a different paper, a Slack discussion, a search query — unambiguously point at this concept, without having to add a project qualifier to disambiguate?"* External-facing counterpart to Criterion 4 (overload risk = internal collision); citability asks whether the name does work *when extracted from project context*. The principle distinguishes three resolution paths when a name fails the test: (a) add a structural qualifier the segment already implies; (b) pivot to a more distinct sibling concept; (c) treat the short form as in-segment shorthand and commit a longer form as the citation handle. A fourth path emerged during the curation pass for adopted-standard cases — *(d) accept the term and discipline first-encounter cite* — applicable when the name is a deliberately adopted standard from an adjacent field and the AAD use is structurally identical to the field's standard meaning. Worked examples that the principle applies to in this cycle: `bias bound` → `Class-3 ambiguity bias bound` (qualifier-add); `specification bound` → `communication bottleneck bound` (sibling-pivot); `multi agent` (accept-as-adopted-standard with cite of Shoham & Leyton-Brown / Stone & Veloso).
