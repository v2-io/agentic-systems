## Goal

The root README is the laggard: PROPOSALS.md Bundle 1 (framework-face reframe) explicitly names it as the remaining unlanded piece. Beyond that one rewrite, the deeper problem is *no mechanism* — each refresh has to be done by hand against drifting source-of-truth files (CHANGELOG, TODO, CLAUDE-2.md), which is the same drift that left the current README at "integration of four disciplines" eighteen months after the convergent reframe.

This proposal does four things together:

1. Establishes **`doc/` as the home for long-lived project process documents** (auditor instructions, naming principles, doc-pipeline templates, etc.), separating them from the working/spike/reflection content that `msc/` was originally for.
2. Establishes a **composable doc pipeline** under `doc/readme/` with Liquid templates, so README.md and README-auditor.md are both *generated* from shared partials plus auto-extracted live sections.
3. Establishes **Findings as a first-class segment section** — each segment optionally carries a `## Findings` block with novelty / impact / casual-curious framing, auto-extracted into FINDINGS.md, which is the canonical curated catalog external readers and the README pipeline both reference.
4. **Sunsets stale duplicate documents**: per-part `README.md` files (only `01-aad-core/README.md` exists; it duplicates the OUTLINE.md preamble) and CLAUDE-2.md (mostly drifted prose; audit-safe content folds into CLAUDE.md, audit-unsafe content lives in its already-existing canonical homes — FINDINGS.md, CHANGELOG.md, TODO.md, PROPOSALS.md, segments themselves).

Doing all four together is what makes the rebuild reliable: composable templates give cheap regeneration, segment-level Findings put the novelty assessment next to the math (so it stays honest and is updated when the math is), the extraction step means README and external readers never drift independently from the segments, and pruning the duplicate documents removes the surfaces where drift was previously accumulating.

## Architecture

### 1. `doc/` directory — long-lived project documentation

`doc/` is the home for **stable process and orientation documents** the project relies on across cycles. The pattern: each generated-document family gets its own subdirectory containing templates and partial sources; standalone process documents live at the `doc/` top level.

```
doc/
  readme/
    README.md.liquid              # composes the public README
    README-auditor.md.liquid      # composes the auditor README
    src/                          # partials for the README templates
      # hand-edited
      _banner.md
      _about.md
      _historical-context.md
      _what-asf-is.md
      _structure.md
      _overview-concepts.md
      _cross-domain-joining.md
      _maturity-gradient.md
      _navigation.md
      _contributing.md
      _auditor-instructions.md    # auditor-only: how to audit, what to NOT read
      # auto-generated (regenerated by bin/extract-*; do not hand-edit)
      _findings-summary.md
      _recent-progress.md
      _known-issues.md
  de-novo-audit-instructions.md   # moved from msc/
  naming-principles.md            # moved from msc/
  ...                             # (additional process docs added over time)
```

**Why the per-doc-family pattern.** Each future generated-document family (paper drafts, conference handouts, talk slides) follows the same shape: `doc/{family}/{template}.liquid` + `doc/{family}/src/` with hand-edited and auto-generated partials side by side. The Liquid `{% include %}` mechanic doesn't care whether a partial is hand-edited or generated; the leading-underscore convention plus a header comment in each generated partial (`<!-- AUTO-GENERATED by bin/extract-X; do not hand-edit -->`) keeps human editors honest.

**FINDINGS.md** lives at the repo root, not under `doc/`, because it is a canonical reader-facing artifact independent of any template — `bin/extract-findings` writes both `FINDINGS.md` (root, full catalog) and `doc/readme/src/_findings-summary.md` (README-shaped condensed) from the same segment walk.

**What moves into `doc/` from `msc/` immediately:**
- `msc/de-novo-audit-instructions.md` → `doc/de-novo-audit-instructions.md`
- `msc/naming-principles.md` → `doc/naming-principles.md`

**What `msc/` retains** (and what `doc/` is *not* for): spikes, brainstorms, reflections, working notes, audit-cycle pending-findings docs, superseded proposals — anything cycle-specific or exploratory. The `doc/` vs `msc/` distinction is *long-lived process documentation* vs *cycle-specific working material*.

**Other candidates to consider later** (not part of this proposal — flagged for a future audit):
- `spikes/INDEX.md` — could be `doc/spikes-index.md`, but it's an index *of* spike content in `msc/` so it's reasonable to leave it there.
- Older `msc/architectural-proposals-2026-04-22.md` and similar dated artifacts are already superseded; belong in `_obs/` if they aren't already.

### 2. Liquid templates

Liquid is the templating choice (per Joseph). Implementation language: **Ruby**, per the project convention that internal process scripts go in Ruby (matching `bin/naming-aggregate.rb`) and Python is reserved for community-facing artifacts (simulations, reproducibility scripts). Existing Python tools (`bin/build`, `bin/lint-md`, `bin/lint-outline`, `bin/md2context`) need no retroactive rewrite — the convention is forward-looking. New internal tools added by this proposal (`bin/build-readme`, `bin/extract-findings`, `bin/extract-recent-progress`, `bin/extract-known-issues`, `bin/refresh-all`) are Ruby.

A template is just an include sequence (Liquid resolves `{% include "X" %}` to `src/X.md` relative to the template):

```liquid
{% include "_banner" %}
{% include "_about" %}
{% include "_historical-context" %}
{% include "_what-asf-is" %}
{% include "_structure" %}
{% include "_overview-concepts" %}
{% include "_cross-domain-joining" %}
{% include "_maturity-gradient" %}

## Novel Results & Findings
{% include "_findings-summary" %}

## Recent Progress
{% include "_recent-progress" %}

## Known Issues & Open Questions
{% include "_known-issues" %}

{% include "_navigation" %}
{% include "_contributing" %}
```

Auditor template omits the three live includes, prepends `_auditor-instructions` after the About:

```liquid
{% include "_banner" %}
{% include "_about" %}
{% include "_auditor-instructions" %}
{% include "_historical-context" %}
{% include "_what-asf-is" %}
{% include "_structure" %}
{% include "_overview-concepts" %}
{% include "_cross-domain-joining" %}
{% include "_maturity-gradient" %}
{% include "_navigation" %}
{% include "_contributing" %}
```

Single source of truth on shared partials; no marker noise in README.md; trivial to add an auditor-specific framing section without touching the public version. Adding a third audience (paper-version, conference-version) later is just another `doc/{family}/` subdirectory.

### 3. Build invocation

```
bin/build-readme            # regenerates README.md and README-auditor.md from doc/readme/
bin/build-readme --check    # verifies on-disk == regenerated; exits non-zero if drift
```

Both files are committed to the repo (so GitHub renders them), but `--check` in a pre-commit hook or CI step catches the case where someone hand-edits a generated file or forgets to regenerate after editing a partial.

Generalization: `bin/build-doc {family}` invokes the same Liquid pipeline against `doc/{family}/`. README is the first user; future families slot in without new tooling.

## Findings as segment-level content

This is the structural move that anchors everything else.

### Schema

Each segment optionally carries a `## Findings` section, sitting between `## Discussion` and `## Working Notes` per FORMAT.md ordering. Header absent ⇒ section absent ⇒ no findings claim from this segment. This matches the existing optional-section discipline (Working Notes is also optional).

```markdown
## Findings

**Novelty:** *[tier]* — {one-sentence headline claim}.

**Impact:** {one short paragraph on what this unlocks, what it closes,
what it would otherwise force; concrete enough that a reader can
evaluate the claim without re-reading the formal expression}.

**For non-specialists:** {2-3 sentences in plain language; what this
would mean to a thoughtful generalist reader}.
```

Tiers reuse the existing claim-tier vocabulary (Exact / Robust qualitative / Heuristic / Conditional). Tier in the Findings section is the *external* tier — it can differ from the Formal Expression's tier when, for instance, a result is technically Conditional but its qualitative content is Robust qualitative for non-specialist purposes; the Findings section says which.

### Why segment-local

- **Proximity to truth.** The novelty claim sits next to the formal expression that backs it. When the math gets sharpened (or softened), the Findings section is in the same diff.
- **Honest novelty.** Most segments will not have a Findings section — that is the correct default. Non-novel segments (definitions, scope statements, supporting derivations) shouldn't claim novelty just to populate a list. Empty by default keeps the catalog focused.
- **Casual/Curious framing where the author can write it.** The "for non-specialists" line is much easier to write while looking at the math than to retrofit later.
- **Composes with existing build infrastructure.** `bin/build` already supports `--strip-working-notes` (H2 section filter); the same pattern extracts H2 `## Findings` sections.

### Extraction

```
bin/extract-findings [outline_file]    # default: top-level OUTLINE.md
```

Walks the outline in canonical order, opens each segment, extracts its `## Findings` block (if present), prepends the segment's slug + Type + section, writes to two outputs:

- **`doc/readme/src/_findings-summary.md`** — README-shaped condensed include: one block per finding, ~4-6 lines each, links to the segment.
- **`FINDINGS.md`** (root) — full canonical catalog: each entry has the headline + impact + casual framing + segment link, ordered by section. This is what external readers reference for "what has ASF actually proved?"

Both are auto-generated. Hand-edit the segment, regenerate, commit.

### Composes with multi-view assembly

Joseph's earlier observation — that OUTLINE.md + build supports assembling alternative views of the theory — extends naturally:

- `bin/build OUTLINE.md --findings-only` → assembled findings-only document for a paper or talk.
- `bin/build OUTLINE.md --strip-findings` → theory-text-only for the formal paper draft.
- Future: per-audience filters (`--audience=academic`, `--audience=engineering`).

The H2-section discipline is the lever; Findings is the most immediately useful new section to add.

## Refresh mechanism

Three layers, in order of automation:

### Layer 1 — manual regeneration (always available)

```
bin/build-readme            # regenerate both READMEs from current partials + generated files
bin/extract-findings        # regenerate findings-summary.md and FINDINGS.md from segments
bin/extract-recent-progress # regenerate recent-progress.md from CHANGELOG (last N entries)
bin/extract-known-issues    # regenerate known-issues.md from TODO + PROPOSALS
```

Each script is small and standalone. A one-liner `bin/refresh-all` chains them.

### Layer 2 — pre-commit / pre-push hook (optional, recommended)

If any of `01-aad-core/src/**`, `02-tst-core/src/**`, `CHANGELOG.md`, `TODO.md`, `PROPOSALS.md`, or `doc/readme/src/**` changed in the staged diff, run `bin/refresh-all` then `bin/build-readme --check`. If `--check` fails (drift), the regenerated files are staged automatically and the commit message gets a trailing `[readme: regenerated]` note.

This keeps the README never-stale-by-construction without imposing thought on the committer.

### Layer 3 — CI verification

CI runs `bin/build-readme --check` and `bin/lint-readme` (link-resolution + slug-existence). Fails the build if anything is out of sync. Belt-and-suspenders for the Layer 2 hook.

### Cycle-close discipline

When a CHANGELOG entry lands:
1. The cycle's segment edits already include any new `## Findings` content (segment-local).
2. The hook auto-regenerates the live sections.
3. The committer sanity-checks the regenerated README in their PR.

No separate "remember to update the README" step.

## General guidelines (condensed)

**Audience tiering.** Sections 1-6 should be readable by a thoughtful generalist. Sections 4, 8, 9 carry the academic load. Sections 5, 7, 11, 12 carry the engineer / contributor load.

**Epistemic discipline.** No hyperbole. Don't undersell. Every Findings entry carries an explicit tier. Mark uncertainty when it exists; mark stability when claiming it.

**Liveness discipline.** Sections 9-11 are auto-generated from canonical source-of-truth files; never hand-edit. Static partials never duplicate live content. Anything that would rot belongs in a live section or gets cut.

**Voice & relic discipline.** Theory-as-it-is, not chronicle-of-changes. Cycle attribution lives in CHANGELOG, not README. No "we recently added," "previously called," "inherited from" except in the Historical Context section. No agent-commentary tone.

**Rendering discipline.** GitHub markdown + Obsidian must both render every section. MathJax `$...$` and `$$...$$` only; no `\begin{align}`. Relative paths for file links; anchor links for slugs. No emojis. Banner image gets alt text.

**Structural discipline.** One H1. H2 for sections; H3 sparingly. Asides → `msc/` or component OUTLINEs.
