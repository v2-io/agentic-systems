## Principles Observations

**Where the principles file felt under-specified:**

1. **Distinguish canonicalize-among-existing-variants from canonicalize-a-concept-name.** The instructions describe "canonicalize" as committing to one name when prose paraphrases it three different ways. But the distinction between "stop paraphrasing an existing canonical name" (conservative — just discipline) and "make this concept-that-has-no-single-name a canonicalized term" (creative — actually naming something) is important. I found myself using canonicalize for both. The second form is closer to name-unnamed-thing but for things that are partially named. A sub-category would sharpen the aggregation.

2. **Weight guidance for the public-API layer is implied but not stated.** The instructions say "rename is expensive" for the public API layer but don't say whether that should express as a +3 keep vs. something else. I defaulted to +3 keep for the framework names (AAD, TST, etc.) because the rationale for NOT changing them is as strong as the rationale for any individual rename. This felt right but was a judgment call.

3. **The add-alias category could benefit from distinguishing "already established but not officially canonicalized" from "proposed new alias."** Several of my add-alias votes are confirming aliases that already appear in NOTATION.md or LEXICON.md (e.g., "satisfaction gap" for $\delta_{\text{sat}}$). These are more about making the alias relationship official than proposing something new. A distinction would help the aggregation route these differently.

**Criteria I used that aren't named in the principles:**

- **Register coherence**: Some names sat in different registers than adjacent names in the theory (clinical-bureaucratic vs. mathematical vs. Greek-vocabulary). I considered whether a name's register created friction — e.g., "scope-ciy-observational-proxy" mixes an acronym (CIY) with a bureaucratic compound (-proxy). This is related to "scope honesty" and "aging potential" but is specifically about the friction between registers within the same codebase.

- **Parallelism as a virtue**: Several "keep" votes were motivated by the fact that a name is the right parallel to an adjacent name (strategic tempo parallels adaptive tempo; communication gain parallels update gain; scope-multi-agent parallels scope-composite-agent). The principles mention "familiarity gradient" but don't explicitly surface parallelism with existing vocabulary as a criterion worth naming.
