## Principles Observations

A few places where the principles file is under-specified, noticed during this review:

**1. Absent: precedence rule when a concept already has a strong prior-art name.**  
Several times I considered a rename and then noticed that the alternative would displace a term the project has explicitly decided to use from an adjacent field (e.g., "chronica" over $\mathcal H$, "sector condition" from control theory, "Markov blanket" from neuroscience/AI). The principles mention "baggage-adoption" as sometimes right but don't specify when the prior-art name is *required* because the project has committed to citing that lineage. A rule like "if the name is adopted with explicit citation and the adoption is stated as a deliberate design choice in FORMAT.md or CLAUDE.md, a rename requires justifying why that lineage citation is better served by a new name" would clarify these cases.

**2. Absent: guidance on the three-ring structure for naming investment.**  
FORMAT.md's three rings (inevitability core / canonical formulations / empirical-heuristic-discussion) have implications for naming: inevitability-core names should be stable and precise (they will be cited as mathematical results); canonical-formulation names can be more expressive (they are modeling choices the reader should understand); empirical-ring names can be more evocative (they are about the world). The principles file doesn't connect naming criteria to segment ring membership. This would help agents prioritize: stabilize inevitability-core names first; invest in evocative names for the third ring where communal imagination compounds most.

**3. Under-specified: when to vote on prose paraphrase vs. slug.**  
For unnamed concepts, I found myself voting on prose vocabulary ("stability-induced myopia," "contraction-over-drift principle") rather than slug forms. The principles file shows how to vote on slugs (#slug-name) but unnamed-concept votes feel different — they're proposing that a concept deserve a name at all, then proposing a form. A two-step vote structure (first: does this deserve crystallization? second: what form?) might be cleaner for unnamed-concept rows.
